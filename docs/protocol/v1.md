# OpenRTC Protocol v1 (Frozen for Beta)

This document defines the OpenRTC WebSocket protocol contract for v1.

## 1. Envelope (All Messages)

All messages use JSON objects with this envelope:

- `t` (string): message type.
- `id` (string): caller-generated correlation identifier.
- `room` (string, optional by type): target room name.
- `event` (string, optional by type): event name.
- `payload` (JSON, optional by type): type-specific body.
- `meta` (object, optional): metadata (trace fields, pagination cursor, etc.).

### Validation and normalization rules

- Non-JSON messages are rejected with `BAD_REQUEST`.
- Unknown top-level fields are rejected.
- `t` and `id` are required for all messages.
- `room` is required for room-scoped messages and must be tenant-prefixed.
- `event` is required only for `EMIT` (client) and `EVENT` (server).
- `payload` is opaque JSON except where type-specific fields are required.
- Envelope size over 20 KB is rejected with `PAYLOAD_TOO_LARGE`.
- Payload size over 16 KB is rejected with `PAYLOAD_TOO_LARGE`.
- `id` is echoed in responses/errors whenever available.

## 2. Client -> Server Message Types

### `JOIN`

Required fields:
- `t = "JOIN"`
- `id`
- `room`

Optional fields:
- `meta.limit` (integer): capped snapshot page size.
- `meta.cursor` (string): pagination cursor.

### `LEAVE`

Required fields:
- `t = "LEAVE"`
- `id`
- `room`

### `EMIT`

Required fields:
- `t = "EMIT"`
- `id`
- `room`
- `event`
- `payload`

### `PRESENCE_SET`

Required fields:
- `t = "PRESENCE_SET"`
- `id`
- `room`
- `payload` (object state)

## 3. Server -> Client Message Types

### `HELLO`

- Sent after successful auth.
- `payload` includes `conn_id` and server metadata.

### `ERROR`

- `payload.code` is a stable machine code.
- `payload.message` is human-readable.
- `payload.id` mirrors the triggering request id when provided.

### `JOINED`

- Confirms room membership.
- Includes snapshot page fields in `payload`:
  - `members`
  - `presence`
  - `next_cursor`

### `LEFT`

- Confirms room leave.

### `EVENT`

- Room event fan-out payload.

### `PRESENCE`

- Presence state update for a room.

## 4. Compatibility

- v1 is non-durable and at-most-once.
- Ordering is guaranteed only per sender connection.
- Reconnect creates a fresh session; no resume window is supported.
